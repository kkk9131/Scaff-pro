概要（実装の狙い・特徴）
概要: 単一ファイルでScreen B UIをリッチに作り込んだモック中心の実装。
概要: Zustand + react-konva + react-three-fiberで2D/3Dキャンバスと参照ビューアまで構成したフルスタックUI実装。
概要: UIコンポーネントライブラリ的なButton/TabsとPlanning用storeを用意し、Screen Bの基本レイアウトを柔軟に組み立てた実装。
概要: TopBar / ToolPalette / CanvasArea / SidePanel の骨組みとZustandストアを用意したが、キャンバス部分はプレースホルダ中心の段階。
概要: Konvaによる2Dグリッド+ズーム/パンとreact-rndの参照ビューアをMainLayoutでまとめた、ライトテーマ寄りの実装。
コード構造・アーキテクチャ
`frontend/src/app/page.tsx` に Screen B を集約した構成。型エイリアスや補助関数で読みやすさを維持しつつも、サイドパネルUIを含め 700 行超が単一ファイルに集中しており、将来的なロジック実装時はコンポーネント分割が必須。Zustand 等の状態管理レイヤは未導入。
`frontend/src/app/page.tsx` で Screen A（案件一覧）、`frontend/src/app/planning/page.tsx` で Screen B を切り分け、さらに `frontend/src/features/planning/*` と `frontend/src/store/planningStore.ts` でビュー/状態/型（`frontend/src/types/index.ts`）を整理。2D Canvas（react-konva）と3Dビュー（@react-three/fiber）を分離した構造で拡張性が高い。
UIコンポーネント層（`frontend/src/components/ui/*`）とドメイン層（`frontend/src/features/planning/*`, `frontend/src/store/planningStore.ts`, `frontend/src/types/index.ts`）を分離。Screen B は `/projects/[id]` ルート配下で `PlanningScreen` を組み立てる構成で、将来のScreen A実装を前提とした設計。ただしトップの `page.tsx` はNextデフォルトのまま残っている。
`frontend/src/store/uiState.ts` にビュー・タブ・ツールの最小限のUI状態のみを集約し、画面は `components/ScreenB/*` に水平分割。骨組みとしてはシンプルでわかりやすいが、キャンバスやデータ構造は未実装で、今後の機能追加を前提にしたプレースホルダ段階。
`frontend/src/components/layout/MainLayout.tsx` を軸に Header / ToolPalette / SidePanel / CanvasArea / ReferenceViewer をスロット的に差し込むレイアウト設計。Konvaベースの `ScaffoldCanvas` はズーム・パンとグリッド描画を担い、React stateでビュー切り替えを行う。状態管理ライブラリは未導入だが、小さな構成で完結している。
UI/UX・ビジュアル面の評価
ダークモード前提のフルスクリーンレイアウトで、ツールパレット / キャンバス / サイドパネルが一体となったハイエンドなモックUI。タイポグラフィやシャドウ、バッジの使い方が丁寧で、高級感のある「設計ツール」らしさが強い。一方で1ページ内に全要素を詰め込んでいるため、レスポンシブや将来の情報量増加には調整が必要。
Screen A の案件一覧はカード+グリッドレイアウトでシンプルかつモダン、Screen B はZustandストア由来の状態と連動したタブ・チャット・数量ビューを備え、UXの流れが明快。ダークトーンとガラス感のあるUI、3Dビューのワイヤーフレーム、参照ビューアのズーム/パンUIなど、要件ドキュメントの意図に最も近い体験が作られている。
グローバルテーマとGlass/Glowユーティリティで全体のトーンを統一しつつ、TabsやButtonコンポーネントでScreen B UIを構成。キャンバスビューはSVG主体のモックだが、建物平面/立面/足場平面/3D/アイソメ/パースのそれぞれにイメージが描かれており、建築系ツールとしてのイメージは掴みやすい。
レイアウト自体はTopBar＋左右パネル＋中央キャンバスの基本形を押さえているが、キャンバスやサイドパネルはテキスト中心のプレースホルダで、視覚的な「Screen B らしさ」はまだ薄い。ダークモード対応はされているものの、デザインディテールは今後の拡張前提。
ライトテーマ前提で、Lucideアイコンとシンプルなパネル構成により柔らかい業務ツール感を出している。一方で要件で求められている「ダークモードかつ高級感のあるインターフェース」とは方向性がやや異なり、背景や配色を調整するとさらに要件に寄せられる。Konvaキャンバス上のグリッドやズーム表示はUX的には良い。
要件適合度・機能実装の観点
Screen B で要求されているタブ（建物平面/立面/足場平面/3D/アイソメ/パース）とサイドパネル（建物・図面/足場条件/数量/チャット/履歴）、左ツールパレット（選択・外周編集・開口線）を一通りUIとしてはカバー。外周や数量はダミーデータだが、ワークフロー全体のイメージは掴める状態まで到達。
2Dキャンバス（アウトライン頂点編集・グリッド・スナップ）と3Dワイヤーフレーム、フローティング参照ビューア、足場条件・数量・チャットタブなど、要件ドキュメントに列挙された要素をUI/状態モデルとしてかなり忠実に落とし込んでいる。Screen A ↔ Screen B の遷移まで構成されており、MVPとしての完成度が最も高い。
CanvasはSVGベースのモックで、Konva/Three等との連携はこれからだが、Screen B 内で必要とされるビュー/タブ/条件入力/チャット/履歴などUI要素は概ね揃っている。Screen A 相当の案件一覧はまだ未実装（`Home` がNext初期画面のまま）であり、全体フローとしては途中段階。
Screen B のレイアウト骨組みとタブ構成はあるものの、キャンバスでの外周編集・足場表示・3D/数量といった実機能はまだ配置されていない。要件に対する「入り口」は用意されたが、機能面ではこれからという印象。
Konvaベースの2Dキャンバスでグリッド/ズーム/パンまで実装されており、参照用図面ビューアもreact-rndで動くため、Screen B の中心機能（土台となるキャンバス操作）は進んでいる。一方でScreen Aや詳細フォーム、3Dビューはロードマップレベルで、UI全体の要件充足度はB案より一段低い。
